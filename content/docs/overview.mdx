Every auth model we've built for the web — OAuth, sessions, API keys — assumes two kinds of actors: a human user and a static application, with predefined scopes. Agents are neither.

They range from ephemeral one-shot tasks to long-running background workers and multi-step systems, calling external services without constant human interaction, sometimes on behalf of a user, sometimes entirely on their own. They are dynamic, non-deterministic, and often do varying tasks over time.

There is no standard way to give an agent an identity, scope its permissions based on its specific tasks, audit its actions, or revoke it without revoking everything.

## Delegated Agents

Every auth model we use (OAuth, API keys, sessions) to share user resources with external systems is, at the limit, user impersonation with scopes. The server sees who authorized the request and what they authorized. It never sees who is actually making it.

When an agent acts on behalf of a user, today it typically uses the user's OAuth token or a shared client credential. That collapses the runtime agent into someone else's identity:

- **No visibility:** The server can't tell which agent made a request. There's no audit trail per agent — every call looks like it came from the user or at best the client application.
- **No scoping:** Every agent gets a static set of permissions that are shared across all agents. You can't give one agent to do one thing and another agent to do another thing.
- **No isolation:** You can't revoke one agent without revoking all of them. A compromised agent means rotating credentials for everything.

**Agent Auth** makes every agent a first-class principal. Each agent registers under a host with its own keypair. Every request is signed with the agent's private key. The server verifies it cryptographically and sees both the user and the specific agent acting for them.

Instead of broad scopes like "write access to accounts," the server grants named capabilities with per-field constraints — `transfer_domestic` capped at $1,000 to a specific account. Each grant is scoped to the individual agent and independently revocable. One agent misbehaves, you revoke that agent. Everything else keeps running.

## Autonomous Agents

Existing auth models tightly couple the human user and the application. When an agent needs to act on its own — without a user in the loop — there is no identity model for it. The agent is forced to pretend to be a human user: opening a browser, solving a CAPTCHA, and clicking through a signup flow just to use a service. There is no standard way for an agent to register itself, prove its identity, or receive capabilities without impersonating a person.

Agent Auth gives each autonomous agent its own identity. No user session, no impersonation. The agent registers directly with its own keypair, authenticates with signed JWTs, and the server attributes every action to the exact agent instance.

## Discovery

There's no standard way for a service to advertise that it supports agents. Every new integration requires hardcoded configuration or a human pointing the agent to the right endpoint.

Agent Auth defines a well-known endpoint and a registry protocol so agents can find and connect to services automatically:

- **By URL** — the agent fetches `/.well-known/agent-auth` from a domain and gets the full configuration.
- **By intent** — the agent describes what it wants to do in natural language, and a registry returns matching services.

## What the specification covers

The full [specification](/specification) defines the complete protocol across these areas:

1. **Identity** — how agents and hosts establish cryptographic identity with Ed25519 keypairs
2. **Registration** — how agents register under a host on a server and receive capability grants
3. **Authentication** — how agents authenticate requests with short-lived signed JWTs
4. **Capabilities** — how servers define, grant, constrain, and revoke capabilities per agent
5. **Approval** — how user consent works for registration, linking, and capability escalation
6. **Discovery** — how agents find services via well-known endpoints and registry search
7. **Lifecycle** — how agents transition between active, suspended, expired, and revoked states
